2. 案例一：大华荧光抓拍相机

抓拍相机就是我们开车经过路口时"闪一下"的那个设备——它抓拍车辆及车内图像作为交通执法的依据。

行业背景

  • 过去：闪光灯强度很大（上千 lux），拍得清楚但晃眼
  • 现在：交规要求闪光灯强度不能超标，补光越来越弱
  • 挑战：低照度下既要看清车内人脸、车牌，又不能过曝

荧光相机的核心技术：AI-ISP

用 AI 技术赋能 ISP（Image Signal Processor，图像信号处理器），在极低照度（80 lux）下实现清晰抓拍。大华将其包装为"银质暗光引擎技术"。

💡 业界宣传 vs 技术本质：

  • 宣传：AI-ISP 算法、机器学习降噪、暗光模型、神经网络压缩至嵌入式芯片
  • 本质：一个完整的画质增强 pipeline，含暗光增强、降噪、人脸恢复等多个模块

3. 车窗增强系统架构

这是杨工详细展开的核心算法案例——一个典型的工业级图像算法系统。

输入问题

❌ 原始问题

  • 夜晚车窗内看不清人像
  • 图像噪点大
  • 白天车窗彩虹膜
  • 车牌过曝

✅ 系统解决

  • 多路 ISP + 针对性参数
  • 深度学习增强模型
  • 人脸检测 + 人脸恢复
  • 仿射变换 + 图像回贴

算法流程

  1. 车窗角点检测：获取车窗四边形的四个角点，确定 ROI
  2. 人脸检测 + 关键点：检测人脸位置和眼睛/鼻子/嘴巴关键点
  3. 仿射变换：将歪头/扭头的人脸矫正到标准位置
  4. 人脸恢复：重点处理人脸区域（客户最关注的部分）
  5. 全图降噪：车窗其他区域做降噪处理
  6. 回贴：将处理后的车窗区域贴回到原始抓拍大图

三个核心难点

难点解决思路
成对数据难以收集 合成噪声（高斯噪声/ELD 噪声模型）、大模型蒸馏、闪光灯/非闪光灯同时拍摄
嵌入式 NPU 算力有限 只能用卷积类简单结构（单帧 200-300ms），复杂算子/大模型无法部署
无成对数据的图像翻译任务（车牌/彩虹膜） CycleGAN + 注意力机制，保证背景一致性的前提下转换前景

4. 算法开发 = 论文→部署全流程

杨工详细拆解了工业界算法落地的完整流程：

完整链路：需求评估 → 算法调研 → 数据收集 → 仿真验证 → 评审 → 算法部署 → 接口联调 → 测试验收 → 现场运维

  1. 需求评估：和领导一起评估可行性、落地产品范围、开发周期、风险点
  2. 算法调研：看论文、开源方法、做 baseline（跟学校里很像）
  3. 数据收集：测试点位自动采集 + 建立测试集
  4. 仿真验证：用 Python/Matlab 跑初始效果
  5. 评审：给领导和项目经理看效果，认可后才推进
  6. 算法部署：Python → C++，PyTorch → ONNX → 各家 NPU 工具链量化
  7. 接口联调：与应用层同事制定接口规范
  8. 测试 + 现场支持：测试验收，技术支持反馈现场问题

💡 与学校的核心差异：学校从模型到论文只要效果好就行；公司里从需求到部署再到运维，每个环节都不可或缺。耗时、算力、Corner Case 处理比论文指标更重要。

5. 更多案例

激光相机：近红外 + 可见光融合

采用分光棱镜将一束光分为两路：一路近红外（夜晚效果好但无颜色），一路可见光（有颜色但有噪声），两路 ISP 处理后融合。同样技术也用在医疗内窥镜（荧光内窥镜）领域。

交通灯涂色算法

夜晚红灯因过曝拍出来是黄色，需要算法"涂红"。流程：粗定位 → YOLO 检测 → 二次确认（区分红/黄）→ 涂色。看起来简单但全国各点位泛化很难。

7. 学校 vs 公司：算法工程师的差异

维度学校（论文）公司（产品）
核心指标PSNR/SSIM 等客观指标客户满意、领导认可
算法耗时一般不在意极为重要（多算法抢 NPU 资源）
Corner Case不太关注占 80% 时间（难处理的数据）
评判标准论文指标提升客户不投诉、领导点头
代码实现Python/Matlab 仿真即可C++ 嵌入式部署、模型量化
开发周期按论文节奏按项目里程碑、Sprint

📌 Corner Case 的重要性："大部分时间其实都在花在这个上面。简单场景很容易就达到了，但重点是要处理那些很难的地方。"——杨熙丞

📝 课后思考

  1. 【业务理解】为什么低照度下车牌容易过曝，而车内人脸却看不清？如果两种区域需要不同的曝光参数，如何在一张照片上同时兼顾？

    参考思路

    车牌反光率高，补光时容易过曝；车内光线弱，需要更多曝光。解决思路：多路 ISP 对不同 ROI 使用不同参数，或后期用 HDR/多帧融合技术。

  2. 【工程思维】论文中提升算法指标（如 PSNR +0.5dB）和公司中解决 Corner Case，哪个在实际产品中更重要？为什么？

    参考答案

    在公司中解决 Corner Case 更重要。因为主流场景大部分算法都能做好，但一个没处理好的极端 case 可能导致客户投诉甚至法律问题。指标提升 0.5dB 在用户感知上几乎无差别。

  3. 【流程理解】一个算法从论文到产品，需要经过哪些环节？哪个环节最容易被低估？

    参考答案

    需求→调研→数据→仿真→评审→部署→联调→测试→运维。部署环节最容易被低估——模型量化带来的精度损失、算子不支持、NPU 算力限制等，都可能导致算法无法落地。

📌 小结

📷 核心业务

荧光抓拍、车窗增强、车牌/彩虹膜处理、红绿灯涂色——都是日常安防场景

论文指标 ≠ 产品效果。耗时、算力、Corner Case 才是产品重点
